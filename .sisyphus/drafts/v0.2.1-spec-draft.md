## 1. Seamless Switch 조사 결과 (2026-04-15)

### 핵심 발견: IDE는 auth 변경 시 LS를 재시작하지 않는다

**근거**: `extension_formatted_latest.js` L44870~L44929 `handleAuthSessionChange`

```
handleAuthSessionChange(e, t):
  1. getOAuthTokenInfo() → 새 accessToken
  2. MetadataProvider.updateApiKey(newToken) → 내부 metadata만 업데이트
  3. LS가 이미 시작됐으면:
     a. client.getStatus({ metadata }) → 기존 LS 연결에 새 metadata
     b. client.getProfileData({ apiKey: newToken }) → 기존 LS에 새 apiKey
  4. updateUserStatus, setAuthStatus 등 UI 업데이트
```

**의미**: LS 프로세스는 계속 살아 있고, 기존 ConnectRPC 연결에 새 apiKey를 담아 요청만 보내면 auth 전달 가능.

### 우리 CLI에서의 3가지 경로

| 경로 | 방식 | 확실성 | 속도 | 구현 난이도 |
|------|------|--------|------|------------|
| **A. Kill → Respawn** | LS 종료 → 새 토큰으로 재시작 | ✅ 확실 | 2~5초 | 낮음 (기존 코드 재사용) |
| **B. 기존 LS에 새 apiKey RPC** | getStatus/getProfileData에 새 apiKey | 🟡 실험 필요 | <100ms | 중간 |
| **C. USS re-push** | SubscribeToUnifiedStateSyncTopic에 새 uss-oauth bytes | 🔴 불확실 | 미정 | 높음 |

### 주인님 통찰 검증 결과

> "Cockpit은 앱을 띄워야 하니까 앱 재시작. 우리는 백그라운드 LS만 띄움. 더 쉬울 것."

**✅ 검증됨.** IDE 코드가 증명: LS 재시작 없이 기존 연결에 새 토큰 전달 가능.
우리의 경로 B가 IDE의 handleAuthSessionChange와 동일한 방식.

### Spec에 반영할 결론

1. **기본 경로**: Full Switch (경로 A, kill+respawn). 확실하고 이미 구현된 코드 재사용.
2. **실험적 경로**: Seamless (경로 B, 기존 LS에 새 apiKey RPC). 실험 후 성공하면 기본 경로로 승격.
3. **경로 C는 NOT NOW**: USS re-push는 불확실성이 높아 v0.2.2+에서 조사.
4. **Live Attach 경로**: IDE가 이미 떠 있으면 IDE의 LS를 그대로 쓰므로, CLI 쪽에서는 auth inject만 하면 됨 (경로 B 불필요).

## 3. 통합 Spec 구조안

```
1. 개요 (auth-only 리팩토링)
2. Feature 1: Auth Login (OAuth + Local Import + import-token)
3. Feature 2: Auth List (quota 진도바 + Cloud Code Quota)
4. Feature 3: Auth Inject (계정 전환 = 토큰 교체)
5. Feature 4: Auto-Rotate (Sticky-Threshold + Wake-up)
6. Feature 5: Seamless Switch (experimental, 조사 결과 기반)
7. Account Store (스키마 + 마이그레이션)
8. 성공조건 (주장+증명 쌍)
9. NOT NOW
10. 구현순서
11. 참조맵
```
